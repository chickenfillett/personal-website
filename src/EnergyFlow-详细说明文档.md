# EnergyFlow 详细说明文档

---

## 第一章 应用概述

### 1.1 产品简介

EnergyFlow 是一款专注于个人精力管理的桌面追踪应用。与传统的情绪记录工具不同，EnergyFlow 将关注点从"情绪"转向"精力"，帮助用户理解自身精力的波动规律、识别消耗因素、发现恢复模式，从而实现更科学的自我管理和生活优化。

应用名称"EnergyFlow"寓意精力的流动——如同河流般有涨有落、有急有缓，而通过持续追踪与深度分析，用户可以更好地驾驭这股流动，在消耗与恢复之间找到平衡点。

### 1.2 设计理念

EnergyFlow 的核心设计理念可以概括为三个关键词：**轻量、深度、安全**。

**轻量**：应用以三种固定界面形态运行，从 48×48 像素的最小化悬浮图标到 480×600 像素的全功能面板，用户可以在不打断工作流的情况下快速记录精力状态。全局快捷键支持一键唤起，记录一次精力状态仅需数秒。

**深度**：应用内置了强大的分析引擎，包括 Pearson 相关性分析、卡方检验、弹性指数计算、周期性检测、语义消耗分析等高级统计方法，能够从看似简单的精力记录中挖掘出深层次的行为模式和健康洞察。

**安全**：所有敏感数据均采用 AES-256-GCM 加密存储，备份文件使用 PKWARE ZIP 2.0 传统加密保护，支持反截屏功能，4位数字密码锁为应用提供基础访问控制，contextBridge 安全 IPC 机制确保前后端通信安全。

### 1.3 迁移背景

EnergyFlow 最初基于 Tauri v2 框架开发，后迁移至 Electron 35.x 平台。迁移的主要原因包括：better-sqlite3 在 Node.js 环境下的原生支持更为成熟稳定；Electron 的 BrowserWindow API 提供了更精细的窗口控制能力（如 setContentProtection 反截屏、setBounds 强制尺寸等）；electron-builder 的打包生态对 Windows 平台支持更完善（Portable EXE、NSIS Setup、MSIX 三种分发格式）。

迁移过程中，窗口拖动逻辑从 Tauri 的 data-tauri-drag-region 迁移至主进程 IPC 处理模式，全局快捷键从 Tauri 的注册机制迁移至 Electron 的 globalShortcut API，数据加密层保持不变，前端 SvelteKit + Svelte 5 代码几乎无需修改。

### 1.4 三种界面形态

EnergyFlow 的界面设计遵循"渐进式展开"原则，提供三种固定的界面形态：

| 形态 | 尺寸 | 用途 |
|------|------|------|
| 最小化模式 | 48×48 px | 悬浮图标，显示当前精力表情，点击快速记录 |
| 中间模式 | 400×400 px | 精简记录界面，快速选择精力等级和标签 |
| 全屏模式 | 480×600 px | 完整功能面板，包含记录、时间线、分析、设置 |

这种设计使得用户可以在日常工作中保持最小化图标悬浮于桌面一角，需要记录时一键展开至中间模式快速记录，需要深入分析时再展开至全屏模式查看详细报告。三种形态之间的切换流畅自然，过渡动画经过精心设计，避免视觉闪烁。

## 第二章 技术架构

### 2.1 技术栈总览

EnergyFlow 的技术栈选型兼顾了开发效率、运行性能和安全可靠性：

| 技术 | 版本 | 用途 |
|------|------|------|
| Electron | 35.x | 桌面应用框架，提供窗口管理、IPC通信、系统API |
| SvelteKit | 5 | 前端应用框架，路由与SSR支持 |
| Svelte | 5 | UI框架，$state/$effect/$derived 响应式系统 |
| better-sqlite3 | - | Node.js SQLite 绑定，同步数据库操作 |
| electron-builder | - | 打包工具，支持多格式输出 |
| Vite | 6 | 构建工具，快速HMR与优化构建 |
| Node.js crypto | - | AES-256-GCM 加密实现 |
| PKWARE ZIP 2.0 | - | 传统加密算法，三密钥流密码 |
| Chart.js | - | 图表库，数据可视化 |

### 2.2 架构分层

EnergyFlow 采用经典的 Electron 分层架构，由以下四层组成：

**主进程层（Main Process）**：运行在 Node.js 环境中，负责窗口生命周期管理、IPC 通信处理、文件系统操作、数据库访问、加密解密等核心功能。主进程是应用的"大脑"，所有敏感操作均在此层执行。

**预加载脚本层（Preload Script）**：作为主进程与渲染进程之间的安全桥梁，通过 contextBridge 暴露受限的 API 接口，确保渲染进程无法直接访问 Node.js API 或 Electron 内部模块。

**渲染进程层（Renderer Process）**：运行 SvelteKit + Svelte 5 应用，负责 UI 渲染和用户交互。通过预加载脚本暴露的 API 与主进程通信，不直接依赖 Node.js 能力。

**数据持久层（Data Layer）**：由 better-sqlite3 驱动的 SQLite 数据库，所有用户数据（记录、设置、贴纸）均存储在本地数据库文件中，不依赖任何远程服务。

### 2.3 通信机制

前后端通信遵循 Electron 安全最佳实践：

1. 渲染进程调用 `window.electronAPI.safeInvoke(channel, ...args)` 发起请求
2. 预加载脚本通过 `ipcRenderer.invoke(channel, ...args)` 转发至主进程
3. 主进程通过 `ipcMain.handle(channel, handler)` 接收并处理请求
4. 处理结果通过 Promise 链路返回至渲染进程

对于事件推送（如 toggle-panel），采用 `ipcRenderer.on` → `contextBridge` → 回调函数的模式，确保主进程可以主动通知渲染进程。

## 第三章 主进程详解

### 3.1 窗口管理

主进程通过 `BrowserWindow` 创建无边框透明窗口，窗口配置如下：

```javascript
{
  frame: false,
  transparent: true,
  resizable: false,
  skipTaskbar: true,
  alwaysOnTop: true,
  hasShadow: false
}
```

三种固定尺寸通过 `setBounds` 方法强制设定，并配合三重保护机制确保窗口尺寸不会被意外改变：

**第一重保护：setMinimumSize / setMaximumSize**。将窗口的最小尺寸和最大尺寸均设置为相同值，从物理层面限制窗口大小调整。

**第二重保护：setBounds 强制大小**。在每次窗口尺寸变化时，立即调用 `setBounds` 将窗口恢复至目标尺寸，确保即使第一重保护被绕过，窗口也会迅速恢复。

**第三重保护：resize 安全网**。监听 `resize` 事件，在事件回调中检测当前窗口尺寸是否与目标尺寸一致，如不一致则立即修正。这层保护主要应对某些 Windows 系统行为（如 DPI 缩放变化）导致的尺寸偏移。

### 3.2 窗口拖动

窗口拖动完全在主进程处理，避免了渲染进程拖动可能导致的性能问题和坐标偏差。拖动流程如下：

1. 渲染进程检测到 header 区域的 `mousedown` 事件，通过 `safeInvoke('window-drag-start', { x, y })` 通知主进程
2. 渲染进程监听 `mousemove` 事件，通过 `safeInvoke('window-drag-move', { x, y })` 持续发送鼠标位置
3. 渲染进程监听 `mouseup` 事件，通过 `safeInvoke('window-drag-end')` 通知主进程拖动结束
4. 主进程在收到 drag-move 时，根据鼠标位移计算新窗口位置并调用 `setBounds` 移动窗口

这种设计将窗口位置计算和设置集中在主进程，渲染进程仅负责捕获鼠标事件和传递坐标，保证了拖动的流畅性和准确性。

### 3.3 全局快捷键

全局快捷键的切换逻辑完全在主进程处理。`sendTogglePanel()` 函数负责切换窗口的显示/隐藏状态：

- 当窗口可见时，隐藏窗口并记录当前窗口位置
- 当窗口隐藏时，显示窗口并恢复至上次位置

快捷键的注册通过 `globalShortcut.register(key, callback)` 实现，注册的快捷键在应用启动时从数据库读取，用户可以在设置界面修改。

值得注意的是，`toggle-panel` 事件虽然仍会从主进程发送至渲染进程，但渲染进程的监听器为空操作。这是因为快捷键逻辑已完全移至主进程处理，保留事件通道仅为兼容性考虑。

### 3.4 HTTP 服务器

主进程启动一个本地 HTTP 服务器，监听 `127.0.0.1:3927`，提供静态文件服务。该服务器用于在开发模式下加载前端资源，同时也为某些需要通过 HTTP 协议访问的资源提供支持（如 Chart.js 的图片导出功能需要同源策略支持）。

服务器仅绑定本地回环地址，不接受外部连接，确保安全性。

### 3.5 IPC 处理器

主进程注册了以下核心 IPC 处理器：

| 通道 | 方向 | 功能 |
|------|------|------|
| window-resize | 双向 | 切换窗口尺寸模式 |
| window-drag-start | 渲染→主 | 开始窗口拖动 |
| window-drag-move | 渲染→主 | 窗口拖动移动 |
| window-drag-end | 渲染→主 | 结束窗口拖动 |
| export_encrypted_backup | 渲染→主 | 导出加密备份 |
| import_encrypted_backup | 渲染→主 | 导入加密备份 |
| toggle-anti-capture | 双向 | 切换反截屏 |
| get-anti-capture | 渲染→主 | 获取反截屏状态 |
| set-autostart | 渲染→主 | 设置开机自启动 |
| auth-* | 渲染→主 | 认证相关操作 |
| records-* | 渲染→主 | 记录CRUD操作 |
| settings-* | 渲染→主 | 设置读写操作 |
| stickers-* | 渲染→主 | 贴纸管理操作 |

### 3.6 反截屏

反截屏功能通过 `mainWindow.setContentProtection(enable)` 实现，该 API 是 Electron 提供的 Windows 平台专属功能，调用后会告知 Windows 窗口管理器将应用窗口标记为"受保护内容"，其他应用无法通过截屏或屏幕录制捕获窗口内容。

该功能仅对 Windows 10 及以上版本有效，在其他平台上为空操作。

### 3.7 开机自启动

通过 `app.setLoginItemSettings({ openAtLogin: enabled })` 实现开机自启动。用户可以在设置界面开关此功能，状态变更通过 IPC 传递至主进程执行。

### 3.8 智能窗口定位

当窗口从隐藏状态恢复显示时，主进程会根据当前窗口位置自动计算新窗口的弹出方向。具体逻辑为：

1. 获取当前窗口的位置和尺寸
2. 计算当前窗口中心点相对于屏幕中心的位置
3. 如果窗口偏左，则向右弹出；偏右则向左弹出
4. 如果窗口偏上，则向下弹出；偏下则向上弹出
5. 确保弹出后的窗口不超出屏幕边界

这种智能定位机制确保了窗口始终在用户视线范围内弹出，不会出现在屏幕外或被其他窗口遮挡。

## 第四章 预加载脚本

### 4.1 安全桥梁

预加载脚本（electron/preload.cjs）是 Electron 安全模型的核心组件。它运行在具有 Node.js 访问权限的特殊上下文中，但通过 `contextBridge.exposeInMainWorld` 仅向渲染进程暴露受限的 API 接口。

### 4.2 暴露的 API

预加载脚本暴露的 API 命名空间为 `window.electronAPI`，包含以下方法：

**通用 IPC 方法**：
- `safeInvoke(channel, ...args)`：安全调用主进程 IPC 处理器，返回 Promise
- `safeSend(channel, ...args)`：安全发送单向消息至主进程
- `safeOn(channel, callback)`：安全监听主进程事件

**对话框方法**：
- `safeDialogSave(options)`：弹出保存文件对话框
- `safeDialogOpen(options)`：弹出打开文件对话框

**文件操作方法**：
- `safeWriteFile(path, data)`：安全写入文件

**系统功能方法**：
- `safeAutostart(enabled)`：设置开机自启动
- `toggleAntiCapture(enable)`：切换反截屏
- `getAntiCapture()`：获取反截屏状态

**窗口拖动方法**：
- `startManualDrag({ x, y })`：开始手动拖动
- `dragMove({ x, y })`：拖动移动
- `dragEnd()`：结束拖动

所有方法均经过参数校验和类型检查，防止恶意注入或非法调用。`safeInvoke` 方法内部对 channel 名称进行白名单校验，仅允许预定义的通道通过。

## 第五章 后端模块

### 5.1 加密模块（crypto.cjs）

#### 5.1.1 AES-256-GCM 加密

EnergyFlow 使用 AES-256-GCM 作为核心加密算法。GCM（Galois/Counter Mode）是一种认证加密模式，同时提供数据机密性和完整性保护。与 CBC 模式相比，GCM 模式无需额外的 HMAC 即可验证数据未被篡改。

加密流程：
1. 使用 PBKDF2 从用户密码派生 256 位密钥，迭代次数为 100,000 次
2. 生成 12 字节随机 IV（初始化向量）
3. 使用 AES-256-GCM 加密明文数据
4. 输出格式：IV（12字节） + Auth Tag（16字节） + 密文

解密流程：
1. 从密文中提取 IV、Auth Tag 和加密数据
2. 使用 PBKDF2 从密码重新派生密钥
3. 使用 AES-256-GCM 解密并验证 Auth Tag
4. Auth Tag 验证失败则抛出异常，表明数据被篡改或密码错误

#### 5.1.2 批量加密/解密

crypto.cjs 提供了批量加密和解密函数，用于备份导入导出场景。批量操作会对每条记录独立生成 IV 和 Auth Tag，确保即使两条记录内容相同，密文也完全不同，防止密文分析攻击。

### 5.2 认证系统（auth.cjs）

#### 5.2.1 密码设置与验证

EnergyFlow 采用 4 位数字密码作为访问控制手段。密码设置流程：

1. 用户输入 4 位数字密码
2. 密码通过 PBKDF2 派生为 256 位密钥后存储密钥的哈希值
3. 原始密码不存储，仅存储派生密钥的哈希

密码验证流程：
1. 用户输入 4 位数字
2. 通过相同的 PBKDF2 参数派生密钥
3. 计算哈希并与存储的哈希比对
4. 匹配则生成 session token

#### 5.2.2 Session 管理

Session 基于时间戳的 token 实现。验证成功后，系统生成一个包含时间戳的 token 并存储在数据库中。后续操作通过比对 token 和时间戳判断 session 是否有效。session 的有效期设计为应用运行期间有效，关闭应用后 session 失效，需重新验证密码。

#### 5.2.3 锁定/解锁逻辑

- 锁定：清除当前 session token，渲染进程显示密码输入界面
- 解锁：验证密码后生成新 session token，恢复正常界面

### 5.3 数据库模块（db.cjs）

#### 5.3.1 数据库初始化

使用 better-sqlite3 创建 SQLite 数据库连接。better-sqlite3 是同步 API，相比异步的 sqlite3 库，在 Electron 主进程中使用更为简洁高效，无需处理回调地狱或 async/await 的复杂性。

数据库文件存储在 Electron 的 `app.getPath('userData')` 目录下，确保数据文件位于用户数据目录，不会因应用更新而丢失。

#### 5.3.2 数据表结构

**records 表**：存储精力记录
- id：主键，自增
- level：精力等级（1-5）
- tags：标签，JSON 数组
- sources：来源，JSON 数组
- status：状态
- emoji：Emoji 标记
- sticker：自定义贴纸
- note：文字备注
- created_at：创建时间戳
- updated_at：更新时间戳

**settings 表**：存储应用设置
- key：设置键名，主键
- value：设置值

**stickers 表**：存储自定义贴纸
- id：主键，自增
- name：贴纸名称
- data：base64 编码的图片数据
- created_at：创建时间戳

#### 5.3.3 自动迁移

db.cjs 实现了自动迁移机制，在每次应用启动时检查数据库 schema 版本，自动执行必要的 ALTER TABLE 或 CREATE TABLE 语句，确保数据库结构始终与当前应用版本匹配。迁移过程是幂等的，重复执行不会产生副作用。

### 5.4 记录管理（records.cjs）

#### 5.4.1 CRUD 操作

records.cjs 提供完整的记录增删改查操作：

- **create**：创建新记录，自动填充 created_at 和 updated_at
- **read**：按 ID 查询单条记录，或按条件查询多条记录
- **update**：更新记录，自动更新 updated_at
- **delete**：按 ID 删除记录

所有数据库操作均使用参数化查询，防止 SQL 注入攻击。

#### 5.4.2 统计分析

getMoodStats 函数是记录管理的核心分析函数，返回以下统计数据：

- **日均精力**：所有记录的精力等级日平均值
- **周均精力**：所有记录的精力等级周平均值
- **时段分布**：将一天分为多个时段（凌晨/早晨/上午/中午/下午/傍晚/晚上/深夜），统计各时段的平均精力等级
- **来源分布**：统计各来源（工作/社交/运动/休息等）的出现频率和对应平均精力
- **状态分布**：统计各状态（专注/分心/放松/紧张等）的出现频率和对应平均精力
- **标签排行**：按出现频率排序的标签列表
- **最高/最低记录**：精力等级最高和最低的记录详情
- **最忙碌日**：记录数量最多的日期
- **因素组合**：最常见的标签/来源/状态组合

#### 5.4.3 PKWARE ZIP 2.0 加密

records.cjs 实现了 PKWARE ZIP 2.0 传统加密算法，用于备份文件的密码保护。该算法是一种三密钥流密码，具体实现如下：

**密钥初始化（pkwareInitKeys）**：
1. 将密码逐字符更新到三个 32 位密钥中
2. key0 和 key1 初始值为特定常量
3. key2 初始值为 0
4. 每个密码字符经过 CRC32 更新 key0，取 key0 高位字节更新 key1，key1 经过特定变换更新 key2

**加密过程（pkwareEncrypt）**：
1. 生成 12 字节加密头：前 11 字节为随机值，第 12 字节为明文 CRC 校验值的高位字节
2. 加密头先经过三密钥流加密
3. 明文数据逐字节加密：每字节先与 key2 高位字节异或，然后更新三个密钥

**解密过程（pkwareDecrypt）**：
1. 解密 12 字节加密头
2. 验证第 12 字节是否与明文 CRC 校验值的高位字节匹配
3. 匹配则密码正确，继续解密数据；不匹配则密码错误

**CRC32 校验**：使用标准 CRC32 算法（多项式 0xEDB88320）计算数据的校验值，用于加密头验证。

#### 5.4.4 导出格式

v4 导出格式：使用 PKWARE ZIP 2.0 密码保护创建 ZIP 文件，ZIP 内含明文 backup.json。用户只需记住一个 4 位密码即可保护整个备份文件。相比 v3 格式（先 AES 加密 JSON 再打包 ZIP），v4 格式的优势在于：标准 ZIP 工具可以识别加密标记，密码错误时能立即提示而非解压后才发现数据损坏。

#### 5.4.5 导入兼容

导入功能支持从 v4 到 v1 的所有历史格式，兼容链路为：v4 → v3 → 明文 JSON → v2 → v1。

- **v4**：PKWARE ZIP 加密 ZIP，输入密码解压后获取 backup.json
- **v3**：ZIP 包含 AES 加密的 JSON，输入密码解密后获取数据
- **v2**：ZIP 包含明文 JSON，无需密码
- **v1**：明文 JSON 文件，直接解析

导入时按优先级依次尝试各格式，首次成功即停止。这种向后兼容设计确保用户无论使用哪个历史版本的备份，都能成功恢复数据。

#### 5.4.6 冲突处理

导入备份时可能遇到记录冲突（相同 ID 或相同时间戳的记录），提供两种冲突处理模式：

- **skip**：跳过冲突记录，保留本地数据
- **overwrite**：用备份数据覆盖本地冲突记录

冲突检测基于记录的 ID 和 created_at 时间戳，确保数据一致性。

### 5.5 设置管理（settings.cjs）

settings.cjs 实现了基于键值对的设置存储系统。所有设置项以 key-value 形式存储在 settings 表中，支持以下操作：

- **get(key)**：获取指定键的值，不存在则返回默认值
- **set(key, value)**：设置指定键的值，存在则更新，不存在则插入
- **getAll()**：获取所有设置项

存储的设置项包括：全局快捷键、主题偏好、语言偏好、反截屏开关等。值统一以字符串形式存储，使用时由前端进行类型转换。

### 5.6 隐私设置（privacy.cjs）

privacy.cjs 专门管理反截屏相关的隐私设置。将隐私设置从通用设置中独立出来，是因为隐私设置涉及系统级 API 调用，需要特殊处理。

主要功能：
- 获取反截屏开关状态
- 设置反截屏开关状态
- 状态变更时通知主进程调用 setContentProtection

### 5.7 贴纸管理（stickers.cjs）

stickers.cjs 管理用户自定义贴纸，支持以下操作：

- **upload(name, imageData)**：上传贴纸，将图片数据以 base64 编码存储在数据库中
- **getAll()**：获取所有贴纸列表
- **delete(id)**：删除指定贴纸

贴纸以 base64 编码存储在 stickers 表的 data 字段中。这种设计使得贴纸数据与记录数据存储在同一个数据库文件中，备份和恢复时自动包含贴纸数据，无需额外处理。

1. 调用 `authStore.lock()` 清除当前 session

FloatingIcon.svelte 是 48×48 像素的悬浮图标组件，是应用最常驻的界面形态。

#### 6.3.1 SVG 人脸图标

图标采用 SVG 绘制的人脸造型，支持 6 种表情状态：

| 表情 | 触发条件 | 视觉效果 |
|------|----------|----------|
| happy | 今日平均精力 ≥ 4 | 笑脸，嘴角上扬 |
| neutral | 今日平均精力 = 3 | 平脸，嘴角平直 |
| sad | 今日平均精力 ≤ 2 | 悲脸，嘴角下垂 |
| excited | 今日平均精力 = 5 | 兴奋脸，眼睛放大 |
| wink | 空闲动画 | 眨眼，一只眼闭合 |
| blink | 空闲动画 | 眨眼，双眼快速闭合 |

表情根据今日所有记录的平均精力等级自动切换，无需用户手动选择。当今日没有记录时，默认显示 neutral 表情。

空闲动画以 8 秒为间隔随机播放 wink 或 blink 表情，使图标保持"活"的感觉。动画持续约 300 毫秒后恢复为当前精力等级对应的表情。空闲动画通过 `setInterval` 定时触发，在组件销毁时清除定时器。

#### 6.3.3 点击行为

图标的点击行为根据认证状态有所不同：

- **未认证状态**（未设置密码或 session 已过期）：点击后调用 `showFull()` 展开至全屏模式，用户可以在全屏模式中输入密码
- **已认证状态**（session 有效）：点击后调用 `showCompact()` 展开至中间模式，用户可以快速记录精力

#### 6.3.4 拖动与点击区分

由于图标同时支持拖动和点击，需要区分用户的意图。实现方式：

1. `mousedown` 事件触发 `startManualDrag`，同时记录鼠标按下位置
2. `mouseup` 事件触发 `dragEnd`，同时计算鼠标位移
3. 如果位移小于阈值（约 5 像素），判定为点击，执行点击逻辑
4. 如果位移超过阈值，判定为拖动，不执行点击逻辑

### 6.4 中间模式（CompactRecord.svelte）

CompactRecord.svelte 是 400×400 像素的精简记录界面，提供快速精力记录功能。它包含精简版的 MoodPicker 组件，去除了完整的标签选择和备注输入功能，仅保留核心的精力等级选择和快速标签。

## 第八章 分析引擎

分析引擎是 EnergyFlow 的核心差异化功能，提供从基础统计到高级洞察的多层次分析能力。

### 8.1 因素相关性分析（correlation.ts）

#### 8.1.1 Pearson 相关系数

Pearson 相关系数衡量两个变量之间的线性相关程度，取值范围为 [-1, 1]。在 EnergyFlow 中，用于计算各因素（标签、来源、状态）与精力等级之间的相关性。

计算公式：

r = Σ((xi - x̄)(yi - ȳ)) / √(Σ(xi - x̄)² × Σ(yi - ȳ)²)

其中 xi 为因素存在标志（0/1），yi 为精力等级。正相关表示该因素与高精力相关，负相关表示与低精力相关。

#### 8.1.2 卡方检验

卡方检验（χ²）用于判断因素与精力等级之间是否存在统计学上的显著关联。计算步骤：

1. 构建列联表：因素存在/不存在 × 高精力/低精力
2. 计算期望频数
3. 计算 χ² 统计量：χ² = Σ((O-E)²/E)
4. 计算 p 值

p 值通过不完全 Gamma 函数计算，EnergyFlow 在 correlation.ts 中自行实现了不完全 Gamma 函数，无需依赖外部统计库。

#### 8.1.3 置信度评估

根据 p 值将相关性置信度分为三级：

| 置信度 | p 值范围 | 含义 |
|--------|----------|------|
| low | p > 0.1 | 相关性不显著，可能为偶然 |
| medium | 0.05 < p ≤ 0.1 | 相关性可能显著，需更多数据 |
| high | p ≤ 0.05 | 相关性显著，可信度高 |

#### 8.1.4 因素排序与洞察生成

根据 Pearson 相关系数对因素进行排序，分为正面因素（正相关）和负面因素（负相关）。基于排序结果生成洞察，最多 8 条，每条洞察包含因素名称、相关系数、置信度和解读文本。

### 8.2 精力分析引擎（energyAnalysis.ts）

精力分析引擎包含多个分析模块，每个模块从不同角度解读用户的精力数据。

#### 8.2.1 弹性指数（Resilience Index）

弹性指数衡量用户从低精力事件中恢复的能力，由两个子指标组成：

**恢复速度**：计算每次低精力事件（等级 ≤ 2）后，精力恢复到中等水平（等级 ≥ 3）所需的时间。恢复越快，弹性指数越高。

**振荡奖励**：检测精力在短时间内的大幅波动（如从等级 2 快速恢复到等级 5），给予额外奖励。这种"弹性反弹"现象表明用户具有强大的恢复能力。

最终弹性指数 = 恢复速度得分 × 0.7 + 振荡奖励得分 × 0.3

#### 8.2.2 消耗敏感度（Drain Sensitivity）

消耗敏感度分析各因素对精力的消耗程度。对于每个因素，计算包含该因素时的平均精力等级与不包含该因素时的平均精力等级之差。差值越大，说明该因素对精力的消耗（或增益）越明显。

结果按消耗程度排序，帮助用户识别最大的精力消耗源和最佳的精力恢复源。

#### 8.2.3 稳定性指数（Stability Index）

稳定性指数通过计算日间精力波动的标准差来衡量精力的稳定程度。标准差越小，说明精力越稳定；标准差越大，说明精力波动越剧烈。

稳定性指数 = 1 - (标准差 / 最大可能标准差)

指数范围为 [0, 1]，1 表示完全稳定，0 表示极度不稳定。

#### 8.2.4 趋势加速度（Trend Acceleration）

趋势加速度采用三段式趋势分析，将时间范围分为前、中、后三段，分别计算各段的精力平均值，然后分析趋势的变化速度：

- 前段趋势：前1/3与中1/3的精力变化
- 后段趋势：中1/3与后1/3的精力变化
- 加速度：后段趋势 - 前段趋势

正加速度表示精力在加速提升，负加速度表示精力在加速下降，零加速度表示趋势稳定。

#### 8.2.5 周期性检测（Periodicity）

周期性检测使用 F 比率方差分析，检测精力数据是否存在周期性模式：

1. 将数据按候选周期（7天、14天、30天）分组
2. 计算组间方差和组内方差
3. F 比率 = 组间方差 / 组内方差
4. F 比率越大，说明周期性越明显

检测结果包括：是否存在周期性、最佳周期长度、周期模式的描述。

#### 8.2.6 异常检测（Anomalies）

异常检测使用 Z-score 方法识别精力数据中的异常值：

1. 计算所有精力记录的均值和标准差
2. 对每条记录计算 Z-score = (xi - μ) / σ
3. |Z-score| > 2 的记录标记为异常

异常记录分为正向异常（精力异常高）和负向异常（精力异常低），帮助用户识别特殊情况。

#### 8.2.7 压力分类（Stress Classification）

压力分类将压力因素分为两类：

**挑战性压力（eustress）**：虽然消耗精力，但有助于成长和提升，如：学习新技能、承担重要项目。这类压力通常伴随积极的情绪和后续的精力恢复。

**阻碍性压力（distress）**：纯粹消耗精力且无积极回报，如：人际冲突、无聊的会议。这类压力通常伴随消极情绪和持续的精力低落。

分类依据来自 SEMANTIC_MAP 中每个因素的 stress_type 标记。

#### 8.2.8 工作生活平衡（Work-Life Balance）

工作生活平衡分析比较工作日与周末的精力差距：

1. 分别计算工作日（周一至周五）和周末（周六、周日）的平均精力
2. 计算差距：工作日精力 - 周末精力
3. 正差距表示工作日精力更高（工作驱动型），负差距表示周末精力更高（休息恢复型）

分析结果帮助用户了解工作对精力的影响，以及是否需要调整工作节奏。

#### 8.2.9 累积负荷（Cumulative Load）

累积负荷通过 7 天滚动平均基线趋势分析精力的长期变化：

1. 计算每日平均精力
2. 对日均值进行 7 天滚动平均
3. 分析滚动平均的趋势方向

上升趋势表示累积负荷在减轻，下降趋势表示累积负荷在加重，持平表示负荷稳定。

#### 8.2.10 恢复质量（Recovery Quality）

恢复质量区分主动恢复和被动恢复：

**主动恢复**：通过积极活动恢复精力，如：运动、社交、冥想。特征是恢复后精力等级较高且稳定。

**被动恢复**：通过减少活动恢复精力，如：休息、睡眠、发呆。特征是恢复后精力等级中等但可能不够持久。

分析结果帮助用户了解哪种恢复方式对自己更有效。

#### 8.2.11 语义洞察（Semantic Insights）

语义洞察基于 SEMANTIC_MAP 对因素进行类别级别的消耗分析：

1. 将因素按 SEMANTIC_MAP 的分类归入不同类别（工作、社交、健康、环境等）
2. 计算各类别的平均消耗程度
3. 识别消耗最大的类别
4. 聚类预警：当多个相关类别同时出现高消耗时，发出预警

SEMANTIC_MAP 包含 63 个因素的分类信息，每个因素具有：类别、压力类型、效价（正面/负面/中性）、关联因素。

#### 8.2.12 Emoji 贡献分析

Emoji 贡献分析基于 EMOJI_VALENCE_MAP，分析不同 Emoji 标记与精力等级的关联：

EMOJI_VALENCE_MAP 包含 34 个 Emoji 的映射信息，每个 Emoji 具有：效价（正面/负面/中性）、唤醒度（高/中/低）、类别。

分析结果展示哪些 Emoji 最常与高精力或低精力关联，帮助用户理解 Emoji 使用模式。

#### 8.2.13 文本洞察（Text Insights）

文本洞察对用户的文字备注进行深度分析：

**关键词情感分析**：基于 KEYWORD_DICTIONARY 识别备注中的关键词，KEYWORD_DICTIONARY 包含 56 个中文关键词，每个关键词具有：类别、效价、LIWC 分类。

**否定词检测**：NEGATION_WORDS 列表包含中文否定词（如"不"、"没"、"无"等），检测否定词可以翻转后续关键词的效价。例如"不开心"中的"不"将"开心"的效价从正面翻转为负面。

**认知扭曲检测**：识别备注中可能存在的认知扭曲模式，如：过度概括（"总是"、"从不"）、灾难化（"完蛋了"、"糟透了"）、非黑即白（"要么...要么..."）等。认知扭曲检测帮助用户意识到可能的不合理思维模式。

### 8.3 语义映射表

#### 8.3.1 SEMANTIC_MAP

SEMANTIC_MAP 是因素语义分析的基础数据结构，包含 63 个因素的详细映射：

每个因素条目包含：
- **name**：因素名称
- **category**：所属类别（work/social/health/environment/emotion/lifestyle）
- **stress_type**：压力类型（challenge/hindrance/neutral）
- **valence**：效价（positive/negative/neutral）
- **related**：关联因素列表

示例：
```
{
  name: "加班",
  category: "work",
  stress_type: "hindrance",
  valence: "negative",
  related: ["压力", "疲惫", "睡眠不足"]
}
```

#### 8.3.2 EMOJI_VALENCE_MAP

EMOJI_VALENCE_MAP 包含 34 个 Emoji 的效价映射：

每个 Emoji 条目包含：
- **emoji**：Emoji 字符
- **valence**：效价（positive/negative/neutral）
- **arousal**：唤醒度（high/medium/low）
- **category**：类别（emotion/activity/nature/food/symbol）

#### 8.3.3 KEYWORD_DICTIONARY

KEYWORD_DICTIONARY 包含 56 个中文关键词的映射：

每个关键词条目包含：
- **word**：关键词
- **category**：所属类别
- **valence**：效价
- **liwc**：LIWC（语言探究与词数计算）分类

### 9.1 分析报告界面（InsightsView.svelte）

InsightsView 是应用最复杂的界面组件，提供多维度、多层次的数据分析报告。

#### 9.1.1 年度像素图（YearPixels）

年度像素图以热力图形式展示全年的精力分布，每个小方块代表一天，颜色深浅代表当日平均精力等级。用户可以直观地看到全年的精力高低起伏，识别出精力低谷期和高峰期。

像素图采用 GitHub 贡献图类似的布局，横轴为周数，纵轴为星期。鼠标悬停显示具体日期和精力数据。

#### 9.1.2 30天概览快照

30天概览快照提供过去 30 天的关键指标汇总：
- 平均精力等级
- 记录次数
- 最常见状态
- 最常见来源
- 精力趋势方向

支持四种日期范围选择：
- **7天**：最近一周的详细分析
- **30天**：最近一个月的综合分析
- **90天**：最近三个月的趋势分析
- **自定义**：用户自选起止日期

不同时间范围会影响分析的深度和可靠性。7天范围提供最详细的分析但数据量较少，90天范围提供最可靠的趋势但细节较少。

#### 9.1.4 三大报告板块

**概览快照板块**：
- 整体精力评分
- 关键指标卡片
- 精力趋势折线图
- 精力分布饼图

**模式发现板块**：
- 时段模式：各时段精力分布柱状图
- 星期模式：周一至周日精力分布
- 消耗敏感度：各因素消耗程度排序
- 压力分类：挑战性 vs 阻碍性压力比例
- 工作生活平衡：工作日 vs 周末精力对比
- 累积负荷：7天滚动平均趋势图
- 恢复质量：主动恢复 vs 被动恢复效果对比
- 周期性检测：检测到的周期模式

- 因素排名：正面因素和负面因素排名
- 场景回放：典型精力场景的重现
- 反思问答：引导用户深入思考的问答

#### 9.1.5 使用指南弹窗

首次进入分析报告界面时，显示使用指南弹窗，介绍各板块的功能和阅读方法。用户可以随时通过帮助按钮重新查看指南。

TimelineView 以时间线形式展示历史精力记录。

#### 9.2.1 筛选功能

**日期筛选**：选择特定日期查看当天的记录。

**类型筛选**：按记录类型过滤，支持以下类型：

#### 9.2.3 记录卡片与详情弹窗

每条记录以卡片形式展示，包含：

### 9.3 数据管理（DataManager.svelte）

DataManager 提供数据的备份和恢复功能。

#### 9.3.1 加密备份导出

导出流程：
1. 用户点击"导出备份"按钮
2. 弹出密码输入弹窗，用户输入 4 位密码
3. 主进程使用 PKWARE ZIP 2.0 算法创建加密 ZIP 文件
4. 弹出保存文件对话框，用户选择保存位置
5. 文件保存完成，显示成功提示

导出文件格式为 `.zip`，内部包含 `backup.json` 文件，JSON 内容为所有记录、设置和贴纸数据的明文序列化。

#### 9.3.2 加密备份导入

导入流程：
1. 用户点击"导入备份"按钮
2. 弹出打开文件对话框，用户选择备份文件
3. 系统尝试解析文件格式（v4 → v3 → 明文 JSON → v2 → v1）
4. 如果文件加密，弹出密码输入弹窗
5. 密码验证成功后，检测记录冲突
6. 如有冲突，弹出冲突选择弹窗（skip/overwrite）
7. 导入完成，显示导入统计（新增/跳过/覆盖的记录数）

#### 9.3.3 HTML 报告导出

除加密备份外，还支持导出 HTML 格式的分析报告。HTML 报告包含：
- 精力趋势图
- 统计摘要
- 关键洞察

HTML 报告为只读格式，不包含原始数据，适合分享给他人查看。

#### 9.3.4 密码输入弹窗

密码输入弹窗为 4 位数字输入界面，每次输入一位数字自动聚焦到下一位，4 位输入完成后自动提交。弹窗支持退格删除和重新输入。

#### 9.3.5 冲突选择弹窗

冲突选择弹窗列出所有冲突记录，每条记录显示本地版本和备份版本的关键信息（时间戳、精力等级），用户可以选择 skip 或 overwrite。也提供"全部跳过"和"全部覆盖"的快捷操作。

#### 9.4.5 标签管理

- 添加新标签/状态/来源

折叠面板，嵌入 DataManager 组件，提供备份导出/导入和 HTML 报告导出功能。

## 第十一章 安全特性

### 11.1 安全架构概览

EnergyFlow 的安全架构遵循"纵深防御"原则，在多个层面实施安全措施：

```
┌─────────────────────────────────────┐
│           应用层安全                  │
│  4位数字密码锁 + Session 管理        │
├─────────────────────────────────────┤
│           通信层安全                  │
│  contextBridge 安全 IPC              │
├─────────────────────────────────────┤
│           数据层安全                  │
│  AES-256-GCM 加密存储               │
├─────────────────────────────────────┤
│           备份层安全                  │
│  PKWARE ZIP 2.0 加密备份            │
├─────────────────────────────────────┤
│           系统层安全                  │
│  反截屏（Content Protection）        │
└─────────────────────────────────────┘
```

### 11.2 AES-256-GCM 数据加密

AES-256-GCM 是当前最先进的对称加密算法之一，被 NIST 推荐用于保护敏感数据。EnergyFlow 使用该算法保护用户的核心数据：

- 密钥长度 256 位，暴力破解在可预见的未来不可行
- GCM 模式同时提供加密和认证，任何篡改都会被检测
- PBKDF2 密钥派生使用 100,000 次迭代，有效抵抗字典攻击和彩虹表攻击
- 每次加密使用随机 IV，相同明文产生不同密文

### 11.3 PKWARE ZIP 2.0 备份加密

PKWARE ZIP 2.0 传统加密虽然安全性不如 AES-256，但具有广泛的工具兼容性。选择该算法作为备份加密方案的原因：

1. 几乎所有 ZIP 工具都支持 PKWARE ZIP 2.0 加密
2. 密码错误时能立即提示，而非解压后才发现数据损坏
3. 4 位数字密码的强度对于备份文件已足够（备份文件通常不会面临在线暴力破解）

加密实现细节：
- 三密钥流密码算法（key0, key1, key2）
- 12 字节加密头，最后字节为 CRC 高位字节校验
- CRC32 校验确保数据完整性

### 11.4 4位数字密码锁

4位数字密码提供基础的应用访问控制，防止他人在用户离开时随意查看精力数据。密码设计为 4 位而非更长，是基于以下考量：

- 应用数据为个人精力记录，不属于高敏感数据
- 密码锁的主要目的是防止"随手翻看"而非"专业攻击"
- 4 位数字在手机/键盘上输入便捷，减少使用摩擦
- 配合反截屏功能，已能满足日常隐私保护需求

### 11.5 反截屏

反截屏功能通过 Windows Content Protection API 实现。启用后，操作系统会将应用窗口标记为"受保护内容"，其他应用无法通过以下方式捕获窗口内容：

- PrintScreen 键截屏
- 第三方截屏工具（如 Snipaste、ShareX）
- 屏幕录制软件（如 OBS）
- 远程桌面工具

该功能仅对 Windows 10 及以上版本有效。在其他平台上，反截屏开关显示为禁用状态。

### 11.6 contextBridge 安全 IPC

Electron 的 contextBridge 是主进程与渲染进程之间通信的安全桥梁。EnergyFlow 的预加载脚本通过 contextBridge 仅暴露预定义的 API 方法，渲染进程无法：

- 直接访问 Node.js API（如 fs、child_process）
- 直接访问 Electron 内部模块（如 ipcRenderer 的原始方法）
- 绕过安全检查调用未授权的 IPC 通道

所有 IPC 通道名称在预加载脚本中进行白名单校验，未在白名单中的通道名称会被拒绝，防止恶意代码注入。

## 第十二章 样式系统

### 12.1 CSS 变量驱动主题

EnergyFlow 的样式系统完全基于 CSS 变量驱动，主题切换通过修改 CSS 变量值实现，无需重新加载样式表。

核心 CSS 变量包括：

**颜色变量**：
- `--bg-primary`：主背景色
- `--bg-secondary`：次背景色
- `--bg-tertiary`：三级背景色
- `--fg-primary`：主前景色（文字）
- `--fg-secondary`：次前景色
- `--accent`：强调色
- `--accent-hover`：强调色悬停态
- `--border`：边框色
- `--shadow`：阴影色

**精力等级颜色**：
- `--level-1`：耗竭色（深红）
- `--level-2`：低落色（橙红）
- `--level-3`：一般色（黄色）
- `--level-4`：良好色（浅绿）
- `--level-5`：充沛色（深绿）

**效果变量**：
- `--blur`：毛玻璃模糊程度
- `--radius`：圆角半径
- `--transition`：过渡动画时长

### 12.2 毛玻璃效果

EnergyFlow 大量使用毛玻璃效果（backdrop-filter: blur）营造现代感和层次感。毛玻璃效果使背景内容透过半透明的前景元素隐约可见，既保持了界面的通透感，又不影响前景内容的可读性。

实现方式：
```css
.glass-panel {
  background: var(--bg-glass);
  backdrop-filter: blur(var(--blur));
  -webkit-backdrop-filter: blur(var(--blur));
  border: 1px solid var(--border-glass);
}
```

毛玻璃效果在低性能设备上可能影响渲染性能，因此提供了降级方案：当检测到设备性能不足时，使用纯色半透明背景替代毛玻璃效果。

### 12.3 字体

应用使用 SF Pro Display 字体，这是 Apple 平台的系统字体，具有优秀的可读性和现代感。在 Windows 平台上，如果 SF Pro Display 不可用，则回退至 Segoe UI，最终回退至系统默认无衬线字体。

字体栈：`'SF Pro Display', 'Segoe UI', -apple-system, BlinkMacSystemFont, sans-serif`

### 12.4 圆角窗口

窗口采用 16px 圆角设计，通过 CSS border-radius 和 Electron 的透明窗口配合实现。圆角窗口使应用看起来更加柔和友好，与 Windows 11 的设计语言保持一致。

实现要点：
- BrowserWindow 设置 `transparent: true`
- 根元素设置 `border-radius: 16px`
- 根元素设置 `overflow: hidden` 裁剪圆角外的内容
- 四个角落的透明区域可穿透点击（由 Electron 透明窗口自动处理）

### 12.5 自定义滚动条

EnergyFlow 自定义了滚动条样式，使其与整体设计风格保持一致：

- 滚动条宽度：6px
- 滑块颜色：半透明前景色
- 滑块圆角：3px
- 悬停时滑块颜色加深
- 轨道透明

自定义滚动条通过 WebKit 的 `::-webkit-scrollbar` 系列伪元素实现。

### 12.6 动画系统

EnergyFlow 的动画系统包含以下核心动画：

**panel-in**：面板展开动画，从缩小状态过渡到展开状态，包含缩放和透明度变化。持续 200ms，使用 ease-out 缓动函数。

**pulse**：脉冲动画，用于悬浮图标的空闲动画。轻微缩放后恢复，持续 300ms。

**hover/active 微交互**：
- 按钮 hover 时轻微放大（scale: 1.02）和颜色变化
- 按钮 active 时轻微缩小（scale: 0.98）
- 卡片 hover 时轻微上浮和阴影加深
- 所有微交互持续 150ms，使用 ease 缓动

动画设计原则：
- 动画时长不超过 300ms，保持界面响应感
- 优先使用 transform 和 opacity 属性，确保 GPU 加速
- 尊重用户的 `prefers-reduced-motion` 设置，在减少动画模式下禁用非必要动画

## 第十三章 数据流与生命周期

### 13.1 应用启动流程

1. Electron 主进程启动，创建 BrowserWindow
2. 加载预加载脚本，建立安全 IPC 桥梁
3. 加载 SvelteKit 前端应用
4. 前端 onMount 初始化：
   - 从数据库加载用户设置（主题、语言、快捷键等）
   - 恢复上次界面状态（面板模式、导航位置）
   - 检查认证状态，如已设置密码则显示密码输入界面
   - 注册全局快捷键
5. 应用就绪，等待用户操作

### 13.2 记录创建流程

1. 用户在 MoodPicker 中选择精力等级
2. 选择标签、来源、状态
3. 可选：添加 Emoji、贴纸、文字备注
4. 点击确认按钮
5. RecordsStore 收集数据，通过 IPC 发送至主进程
6. 主进程将数据写入 SQLite 数据库
7. 返回新记录 ID
8. RecordsStore 更新本地缓存
9. 界面显示记录成功提示
10. 悬浮图标表情更新

### 13.3 分析报告生成流程

1. 用户切换至分析报告视图
2. InsightsView 根据当前日期范围请求统计数据
3. RecordsStore 通过 IPC 调用主进程的 getMoodStats
4. 主进程查询数据库，计算统计数据
5. 返回统计数据至前端
6. 前端调用 correlation.ts 和 energyAnalysis.ts 进行深度分析
7. 生成分析结果和洞察
8. 渲染分析报告界面

### 13.4 备份导出流程

1. 用户在设置中点击"导出备份"
2. 弹出密码输入弹窗，用户输入 4 位密码
3. 前端通过 IPC 调用主进程的 export_encrypted_backup
4. 主进程从数据库读取所有数据
5. 序列化为 JSON 格式
6. 使用 PKWARE ZIP 2.0 算法创建加密 ZIP 文件
7. 弹出保存文件对话框
8. 用户选择保存位置
9. 写入文件
10. 返回成功结果
11. 前端显示成功提示

### 13.5 备份导入流程

1. 用户在设置中点击"导入备份"
2. 弹出打开文件对话框，用户选择备份文件
3. 前端通过 IPC 调用主进程的 import_encrypted_backup
4. 主进程读取文件内容
5. 按兼容链路尝试解析（v4 → v3 → 明文 JSON → v2 → v1）
6. 如果需要密码，返回密码请求
7. 前端弹出密码输入弹窗
8. 用户输入密码，通过 IPC 发送至主进程
9. 主进程使用密码解密文件
10. 解析 JSON 数据
11. 检测记录冲突
12. 如有冲突，返回冲突列表
13. 前端弹出冲突选择弹窗
14. 用户选择冲突处理模式
15. 通过 IPC 发送冲突处理结果
16. 主进程执行导入操作
17. 返回导入统计
18. 前端显示导入结果

1. **后端**：在 electron/ 目录下创建或修改模块，添加 IPC 处理器
2. **预加载**：在 preload.cjs 中暴露新的 API 方法
3. **前端**：在 src/lib/ 下创建或修改组件和 Store
4. **样式**：使用 CSS 变量确保主题兼容性
5. **测试**：手动测试功能，验证边界情况
6. **打包**：运行 `npm run build` 生成安装包

### 14.4 代码规范

- 主进程代码使用 CommonJS 模块格式（.cjs 扩展名）
- 前端代码使用 ES Module 格式（.ts / .svelte 扩展名）
- Svelte 5 响应式：使用 $state 声明响应式状态，$effect 声明副作用，$derived 声明派生值
- CSS 使用全局变量，不使用 CSS-in-JS
- 组件命名使用 PascalCase，Store 命名使用 camelCase

### 14.5 调试技巧

**主进程调试**：在 package.json 的启动脚本中添加 `--inspect=5858` 参数，然后使用 Chrome DevTools 连接 `localhost:5858` 进行调试。

**渲染进程调试**：开发模式下按 Ctrl+Shift+I 打开 DevTools。生产模式下可通过命令行参数 `--devtools` 启用。

**数据库调试**：使用 DB Browser for SQLite 直接打开数据库文件查看数据。数据库文件路径：`%APPDATA%/energyflow/energyflow.db`。

**IPC 调试**：在主进程的 ipcMain.handle 回调中添加 console.log，或在预加载脚本的 safeInvoke 中添加日志。

## 第十五章 常见问题与故障排除

### 15.1 窗口相关问题

**Q：窗口无法拖动**
A：确保在 header 区域（顶部导航栏）按住鼠标拖动，而非在内容区域。窗口拖动仅在 header 区域生效。

**Q：窗口尺寸异常**
A：三重保护机制会自动修正窗口尺寸。如果问题持续，尝试重启应用。DPI 缩放变化可能导致短暂的尺寸异常。

**Q：窗口出现在屏幕外**
A：智能窗口定位会自动将窗口移回屏幕内。如果问题持续，删除设置中的窗口位置数据，应用将使用默认位置。

### 15.2 快捷键相关问题

**Q：快捷键不生效**
A：检查快捷键是否被其他应用占用。全局快捷键具有系统级优先级，如果其他应用已注册相同快捷键，后注册的应用将无法使用。

**Q：快捷键注册失败**
A：某些快捷键组合被系统保留（如 Ctrl+Alt+Delete），无法注册。请在设置中选择其他快捷键组合。

### 15.3 数据相关问题

**Q：数据库文件损坏**
A：SQLite 具有内置的损坏检测机制。如果数据库损坏，应用会尝试自动修复。修复失败时，使用备份文件恢复数据。

**Q：备份导入失败**
A：确认备份文件格式正确。应用支持 v1 至 v4 所有格式的备份文件。如果密码错误，v4 格式会在解密时立即提示。

**Q：导入后数据丢失**
A：检查冲突处理模式。如果选择了"skip"模式，冲突记录会被跳过，不会覆盖本地数据。尝试使用"overwrite"模式重新导入。

### 15.4 性能相关问题

**Q：应用启动慢**
A：首次启动需要初始化数据库和加载前端资源。后续启动会利用缓存，速度更快。如果持续缓慢，检查数据库文件大小，大量记录可能影响启动速度。

**Q：分析报告加载慢**
A：分析引擎的计算复杂度与记录数量相关。90 天范围的分析可能需要数秒。建议使用 30 天范围获取更快的分析结果。

### 15.5 安全相关问题

**Q：忘记密码**
A：应用密码无法恢复，因为密码哈希不可逆。但用户数据不会丢失，可以通过以下方式重置：
1. 关闭应用
2. 删除数据库中的密码哈希记录
3. 重新启动应用，设置新密码

**Q：反截屏不生效**
A：确认运行在 Windows 10+ 平台，且已在设置中开启反截屏功能。某些第三方截屏工具可能绕过 Content Protection。
